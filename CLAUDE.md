# Refactor: company-only Postgres pipeline, dockerized, with judge TUI


## Context

Today the pipeline runs two parallel discovery streams (the 33-row `sources.csv` site-stream + Google News per top-150 B3 ticker), writes JSONL + per-day CSV files, mirrors a subset into a SQLite DB, and is glued together by a FastAPI app with APScheduler that the React skeleton talks to. After two months of use the site-stream produces a lot of macro noise that doesn't match any tracked company, the dual-write to JSONL/CSV/SQLite is fragile (one extract crash = lost batch), `sources.csv` lives outside the DB so adding a publisher is a code change, and there's no human-judging loop to sanity-check sentiment.

This refactor keeps the company-stream only, makes Postgres the single source of truth, deletes `backend/api/` (the React UI will be rebuilt against a new API in a later refactor — out of scope here), packages everything in Docker Compose with a cron-driven daily run, and adds a one-script terminal TUI for judging articles.

## Locked decisions (from user)

- DB: Postgres 16, local-only via Docker, named volume `pgdata`
- Site-stream + `sources.csv` deleted; `sources.csv` content seeds a new `publishers` table
- Existing data abandoned: no migration of `data/raw/raw_articles.jsonl`, `data/news/*.csv`, or `data/app.db` — fresh DB
- `backend/api/` (FastAPI + SQLAlchemy + SQLite + APScheduler) deleted entirely; web UI rebuild deferred
- Daily run: separate `cron` service in compose runs `python -m finance_news.pipeline run` once/day at 23:50 America/Sao_Paulo
- CLI shims removed; daily/manual triggers go through `pipeline.py`. A Makefile wraps common ops (build, up, ingest, judge, etc.) for dev convenience
- Judging: terminal TUI (readchar) writing to a separate `judgments` table that supports re-judging
- Layout: `backend/pipeline/` → top-level `finance_news/` package; `backend/` dir disappears; `backend/scripts/` → top-level `scripts/`

## Architecture (Docker)

- `db` service: `postgres:16-alpine`, named volume `pgdata`, healthcheck (`pg_isready`), exposes `5432` to host so a host-side `psql` works
- `app` service: `python:3.11-slim`, build deps for spaCy/torch, `pt_core_news_lg` baked into the image, `HF_HOME=/hf_cache` on a named volume `hf_cache` so model weights survive image rebuilds, repo bind-mounted at `/app`, runs as non-root UID 1000, `tini` as PID 1, default command is `sleep infinity` (used as a dev shell — `make ingest`/`make judge` exec into it)
- `cron` service: same image as `app`, command runs a tiny scheduler loop (or `supercronic` with a single-line crontab `50 23 * * * cd /app && python -m finance_news.pipeline run`), `TZ=America/Sao_Paulo`, depends on `db` healthy

## Schema — `migrations/001_init.sql`

```sql
publishers(
  hostname TEXT PRIMARY KEY,        -- e.g. "valor.globo.com"
  display_name TEXT NOT NULL,
  founded_year INT,
  ownership TEXT,                   -- "Privada"|"Pública" from sources.csv; NULL otherwise
  homepage TEXT,
  affiliations TEXT[] DEFAULT '{}', -- from analysis.PUBLISHER_AFFILIATIONS
  notes TEXT
);

companies(
  ticker_root TEXT PRIMARY KEY,     -- "PETR"
  ticker TEXT NOT NULL,             -- "PETR4"
  short_name TEXT, long_name TEXT,
  sector TEXT,
  market_cap BIGINT,
  fetched_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

articles(
  url TEXT PRIMARY KEY,
  title TEXT, text TEXT, author TEXT,
  site TEXT, hostname TEXT,
  published_at TIMESTAMPTZ,
  source_ticker TEXT REFERENCES companies(ticker_root),
  fetched_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  -- analysis (NULL until extract):
  sentiment TEXT, sentiment_score REAL,
  subjects TEXT[], companies_ner TEXT[], persons TEXT[],
  countries TEXT[], currencies TEXT[], matched_tickers TEXT[],
  conflicts TEXT[], summary TEXT,
  extracted_at TIMESTAMPTZ
);
CREATE INDEX articles_published_at_idx ON articles(published_at DESC);
CREATE INDEX articles_pending_extract_idx ON articles(published_at) WHERE sentiment IS NULL;
CREATE INDEX articles_source_ticker_idx ON articles(source_ticker);
CREATE INDEX articles_matched_tickers_gin ON articles USING GIN(matched_tickers);

judgments(
  id BIGSERIAL PRIMARY KEY,
  article_url TEXT NOT NULL REFERENCES articles(url) ON DELETE CASCADE,
  judge TEXT NOT NULL,
  label TEXT NOT NULL CHECK (label IN ('positive','neutral','negative','skip','bad_match')),
  notes TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX judgments_article_idx ON judgments(article_url);

runs(
  id BIGSERIAL PRIMARY KEY,
  kind TEXT NOT NULL,                -- "ingest"|"extract"|"full"
  started_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  finished_at TIMESTAMPTZ,
  status TEXT NOT NULL,              -- "running"|"ok"|"error"
  n_fetched INT, n_extracted INT,
  error TEXT
);

schema_migrations(version TEXT PRIMARY KEY, applied_at TIMESTAMPTZ NOT NULL DEFAULT now());
```

All arrays are `TEXT[]` (not JSON). All access goes through psycopg3 (`psycopg[binary]>=3.1`). No ORM. Only parameterized SQL.

## Rules (per user)
- Less code is better than more code.
- No fallback mechanisms — they hide real failures.
- Rewrite existing components over adding new ones.
- Flag obsolete files to keep the codebase lightweight.
- Avoid race conditions at all costs.
- One stage per turn. Stop after each and summarize.
- Don't touch files outside the current stage's scope.
- Don't run `make build` or `make up` — user verifies each stage manually.
- Real tabs in the Makefile.
- All SQL parameterized; no string interpolation.
- Comment only non-obvious decisions worth second-guessing in three months.
