# Per-company day view (local DeepSeek summary + daily candle chart) — staged

## Context

Today the report page shows aggregate dashboards for a chosen day; there's no way to drill into a single company. We're adding a third tab **Empresa** to the report view. For any processed day the user picks a company and sees:

1. A summary of the **good and bad points** drawn from that day's articles about the company, generated by a **locally-hosted DeepSeek model**.
2. A **daily candle chart** of the stock's price ±10 trading days around the selected day, with the selected day highlighted.

Locked decisions (from prior Q&A):
- **Candle scope**: daily OHLC, ±10 trading days around the selected date (yfinance daily bars are free and unlimited; intraday history isn't reliably available for arbitrary historical days).
- **LLM timing**: eager — generated during the daily extract phase for the top‑N companies of the day. Lazy regeneration is unnecessary because the underlying articles are immutable once a day is processed.
- **LLM runtime**: free remote provider over an OpenAI-compatible endpoint. Default: Groq Cloud's `llama-3.3-70b-versatile` (free tier, ~30 req/min — well above our ~20 calls/day budget). The same code can fall back to Ollama, OpenRouter, or the real DeepSeek API by flipping `LLM_BASE_URL` / `LLM_MODEL` / `LLM_API_KEY` env vars.

## Execution — one stage per turn (stop after each, wait for verify)

### Stage 1 — DB schema + helpers (BE only)

Two new tables, plus the read/write helpers everything else will lean on.

- `migrations/003_company_summaries_and_ohlc.sql` (new):
  ```sql
  CREATE TABLE IF NOT EXISTS company_day_summaries (
      ticker_root   TEXT NOT NULL,
      summary_date  DATE NOT NULL,
      good_points   TEXT[] NOT NULL,
      bad_points    TEXT[] NOT NULL,
      article_count INT  NOT NULL,
      model         TEXT NOT NULL,
      created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
      PRIMARY KEY (ticker_root, summary_date)
  );
  CREATE INDEX IF NOT EXISTS idx_summaries_date
      ON company_day_summaries(summary_date);

  CREATE TABLE IF NOT EXISTS stock_ohlc (
      ticker_root  TEXT NOT NULL,
      bar_date     DATE NOT NULL,
      open         NUMERIC(12,4) NOT NULL,
      high         NUMERIC(12,4) NOT NULL,
      low          NUMERIC(12,4) NOT NULL,
      close        NUMERIC(12,4) NOT NULL,
      volume       BIGINT,
      fetched_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
      PRIMARY KEY (ticker_root, bar_date)
  );
  ```
- [finance_news/store/db.py](finance_news/store/db.py) — append helpers (keyword-only, mirroring existing style):
  - `upsert_company_summary(conn, *, ticker_root, summary_date, good, bad, article_count, model)`
  - `fetch_company_summary(conn, *, ticker_root, summary_date) -> dict | None`
  - `fetch_articles_for_company(conn, *, ticker_root, day) -> list[dict]` (`WHERE %s = ANY(matched_tickers) AND (published_at AT TIME ZONE 'America/Sao_Paulo')::date = %s`, ordered by `sentiment_score DESC NULLS LAST, published_at DESC`).
  - `upsert_ohlc(conn, *, ticker_root, bars)` — `executemany` with `ON CONFLICT (ticker_root, bar_date) DO UPDATE SET ...`.
  - `fetch_ohlc_range(conn, *, ticker_root, start, end) -> list[dict]`.

**Verify**:
- `python scripts/migrate.py` adds both tables; re-run is a no-op.
- Quick psql sanity: `INSERT … company_day_summaries`, then `fetch_company_summary` round-trips. Pipeline still boots (`docker compose restart app && curl /api/health` → 200).

### Stage 3 — Eager summarization step wired into the pipeline (BE)

Make the daily pipeline produce summaries automatically.

- `finance_news/summaries.py` (new) — `run_summaries(target_date, top_n=20, progress=None) -> int`:
  1. `db.fetch_articles_for_date(conn, target_date)`.
  2. `Counter` of `ticker_root` from each row's `matched_tickers`. Take top‑N by article count.
  3. Idempotency: skip tickers already in `company_day_summaries(ticker_root, target_date)`.
  4. Resolve `name` from `companies.short_name`; gather article tuples; call `llm_summary.summarize_company_day`; on success `db.upsert_company_summary(...)`. Failure → log + continue.
  5. Emit `progress("summarize", current, total)` so the existing SSE stepper picks it up unchanged.
- [finance_news/pipeline.py](finance_news/pipeline.py): `run_full(...)` calls `run_summaries(target_date, progress=progress)` after `run_extract`. Add a thin `run_summarize` symmetric to `run_ingest` / `run_extract` so [api.py](finance_news/api.py)'s `kind` switch can run it standalone (used to backfill historical days without re-ingesting).

**Verify**:
- `docker compose exec app python -m finance_news.pipeline --date <day>` finishes; `SELECT ticker_root, array_length(good_points, 1) FROM company_day_summaries WHERE summary_date = '<day>';` returns ≤ top‑N rows with non-empty arrays.
- Re-run on the same date: zero new rows (idempotent).
- Stop the `ollama` service, run again on a fresh date: pipeline still succeeds, log shows the warning, no rows added.

### Stage 4 — Stock OHLC module with lazy caching (BE)

- [requirements.txt](requirements.txt): add `yfinance`.
- `finance_news/stocks.py` (new) — `fetch_ohlc_window(conn, ticker_root, day, span=10) -> list[Bar]`:
  - Resolve B3 symbol: `companies.ticker` if available else `<ticker_root>` + `.SA`.
  - Compute calendar window: `day - 21 days`, `day + 21 days` (≫ ±10 trading days, covers holidays & weekends).
  - Cache check via `db.fetch_ohlc_range`. If the window is dense enough, return the cached subset.
  - Else `yf.Ticker(...).history(start=..., end=...)` → normalize rows to `(bar_date, open, high, low, close, volume)` → `db.upsert_ohlc(...)`.
  - Trim returned list to ±10 trading days around `day` (sort, locate selected, take 10 either side).

**Verify**:
- REPL inside app: `from finance_news.stocks import fetch_ohlc_window; fetch_ohlc_window(conn, 'PETR', date(2026,4,29))` returns ~21 bars.
- Subsequent call returns in <50 ms (cache hit).
- Unknown ticker returns `[]` (not an exception).

### Stage 5 — API endpoints (BE)

Mirror the `/api/reports/{date}` style.

- `GET /api/companies/{ticker_root}/summary/{date}` →
  ```ts
  {
    ticker, name, date,
    good: string[], bad: string[],
    articleCount, model,
    articles: Array<{ url, title, site, sentiment, sentimentScore }>  // ≤10
  }
  ```
  - 400 bad date; 404 if no `company_day_summaries` row.
  - Article evidence from `db.fetch_articles_for_company`, capped at 10.

- `GET /api/stocks/{ticker_root}/ohlc/{date}` →
  ```ts
  {
    ticker, selectedDate,
    bars: Array<{ date, open, high, low, close, volume }>  // ~21
  }
  ```
  - 400 bad date. Empty `bars[]` (still 200) when yfinance has nothing.

## Rules

- One stage per turn. Stop after each and summarize.
- Don't touch files outside the current stage's scope.
- Every stage leaves the app in a runnable state — no broken intermediate commits.
- All chart fills/strokes use `hsl(var(--…))` strings or `SENTIMENT_COLORS` constants — palette stays single-sourced in [index.css](web/src/index.css).
- LLM client must soft-fail on Ollama outage; the rest of the pipeline cannot regress because of summarization.
